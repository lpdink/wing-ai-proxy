# 需求

我们正在构建一个良好架构的Go项目 用于LLM的代理，客户端会通过 HTTP请求OpenAPI兼容格式来请求我们 但是将来也有可能是通过 比如说Anthropic兼容协议来请求我们 这就意味着我们要做好处理 要面向未来编程 但是对于第一版来说 我们只支持OpenAI兼容格式来请求我们 也就是说 对于下游协议来说 我们暴露获得模型的那个接口/models和CHAT接口就可以了。

然后我们做什么呢 我们就是透传，客户端的所有请求我们都透传 包括什么header呀 什么request的body呀 都透传给上游。

然后我们要做好流式SSE的处理 也就是说客户端是流式的 我们就要流式的返回给他客户端是非流的 我们就非流的返回给他。

由于我们是一个中转 所以我们可以对整个过程进行审计 我们要审计的 比如说他这个请求是什么时候发起的 然后首包时间是多少 用的什么模型，输入输出tokens是多少，缓存命中了多少。然后呃 完整的请求和这个响应，以及其他顺手可以审计的性能指标。存储到SQLite里面。存储也面向未来设计，也支持存储到pg里面。但是第一个版本，我们存储到sqlite里面。

我们要支持配置文件的热加载。
我们的配置文件大概是这个样子的，我稍微描述了一下意思，具体语法可能不太对，看着修复一下。

```yaml
# ~/.wing-ai-proxy/
host: 127.0.0.1
prot: 39998
virtual-api-keys:
    - sk-abcdefg (我们也不检查它的格式，对得上就放过)
providers:
    dashscope:
        - type: openai
        - base_url: xxxx
        - api_key: xxxx
        - models:
            - show_name: real_name 
            # show name 比如说，ds-v4-flash， real_name: deepseek-v4-flash，这样客户端可以少打几个字。
            - show_name2: real_name2
    openai:
        - type: openai
        - base_url: xxxx
        - api_key: xxxx
```

我们发给上游的方法由type决定，在将来，也许我们的上游提供商提供的是Anthropic协议，我们也就要支持Anthropic格式，不过现在不着急，第一个版本我们只支持openai格式。

当有多个provider提供了相同的show_name的时候，我们怎么做？我们打个warning，然后始终走第一个！
（终归结底 这个项目目前暂时只是让我一个人去用的 我是害怕我自己手滑配错了 然后破坏缓存命中 因为很多上游服务提供商 他是按照账户级别去隔离的 如果一不小心我们随机发到第二个上面 那就没有缓存命中了 我就寄了）

我们的项目要良好架构，良好抽象和拆分，不要都堆在一个文件里。
我们要通过Go的静态检查，go vet，创建一个Makefile文件，支持构建测试和静态检查 以及代码格式化。
我们的项目要有良好的可观测性，日志要打够 然后什么请求ID什么的也得给我加上，当然这个请求ID 呃 不会传给上游 也不会传给下游。

# 功能点/测试

1. /models 获取模型列表，显示的当然是show_name的列表啦
2. openai chat接口：流式和非流调用chat接口都要成功。
3. 请求透传，我们完全不修改请求本身，header，body等一切信息都透传
4. 审计
我们要支持比较多的审计 能回答很多问题啊 这就意味着我们在存那个数据库的时候 要想清楚这个数据库的Schema 我来说一下我们要审计什么吧 常见的一些审计比如说：
哪个虚拟api key，走的哪个真实的provider，在哪段时间范围内，用了哪些模型，请求了多少次？用了多少输入tokens？多少输出tokens？多少缓存命中？某个模型的平均首包是多少？TPS是多少？（tokens/s）模型下发了多少个tool call？调用的工具名，
我们要完整保存所有的请求和响应信息，即使它是SSE的，我们也要攒后保存完整的请求和响应。

我们可以通过sql查询来支持以上审计。
不怎么用考虑api key的泄露等问题，尤其是虚拟api key，完全是我们随便修改config.yaml，随时生效，删除后随时吊销。
这里也没有多用户的概念，一个虚拟api key就有点像我们的一个用户。
对config.yaml的修改要能热重载。
